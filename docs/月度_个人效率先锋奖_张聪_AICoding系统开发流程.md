## Slide 4 — 一、专项概述

**布局**：左文 + 右 OKR 表格

### 专项背景及意义

AICoding 正在**根本性改变新系统开发方式**：从「手写样板代码 + 反复对齐需求」转向「规范驱动 + Agent 编排 + 全栈一键交付」。实践表明，一条完整链路——需求澄清 → 方案设计 → 拆解计划 → 编码实现 → 测试验证 → 文档交付——可在**数天至数周**内完成从 0 到 1 的可上线系统。

目前已用 AICoding 落地三个线上平台：**大模型场景管理系统**、**代码智能解析平台**、**智能体人机协作平台**。它们均非「一句话生成 Demo」，而是经过多轮产品迭代、具备前后端 + 数据库 + 可持续维护能力的**完整应用**。

然而，AICoding 提速的同时带来三类新挑战：

1. **「快写快丢」风险**：跳过设计直接写代码，会话结束即失忆，系统难以迭代维护
2. **需求与代码脱节**：口头改需求、验收场景模糊，AI 产出与产品预期偏差大
3. **团队无法复制**：个人会话经验未沉淀为规范、Agent、Skill，新人接入成本高

本专项目标是：构建一套**规范驱动、可复用、可归档**的 AICoding 系统开发流程，让新系统**快速上线**且**可持续迭代**。

### 整体目标及达成情况

建立「**Spec 为真理源 + @网站开发 Agent 编排 + 确认节点门禁**」的全栈交付体系：

- **产品侧**：写清 MVP 与验收场景，在 design.md 确认前不写代码
- **开发侧**：`@网站开发` 按 brainstorming → writing-plans → executing-plans 链路自动加载 Skill
- **团队侧**：`spec/` 目录沉淀需求 / 设计 / 能力规格，变更归档后可追溯、可复用

## Slide 7 — 二、专项成果 · 规范驱动目录体系

**布局**：成果展示（左说明 + 右目录结构图）

### 标题

二、专项成果 - 成果展示 | 规范驱动 spec/ 目录体系

### 核心设计

`spec/` 目录是需求、设计与系统能力的**单一事实来源**。代码在 `src/`，但「做什么、做到什么程度、怎么验收」以 spec 为准。

### 目录职责

| 路径 | 用途 | 是否真理源 | 产品关注点 |
|------|------|-----------|-----------|
| `spec/specs/` | 当前已上线系统能力 | ✅ 是 | 查「系统现在能做什么」 |
| `spec/changes/` | 进行中的需求迭代 | 提案工作区 | 评审 proposal / design / tasks |
| `spec/ME2AI/` | 人类 → AI 原始输入 | ❌ 否 | 写需求、反馈、背景材料 |
| `spec/AI2AI/` | AI 中间产物草稿 | ❌ 否 | 确认前可忽略；确认后迁入 changes |
| `spec/project.md` | 项目上下文 | 项目记忆 | 技术栈、术语表、Agent 清单 |

### 变更目录标准结构

每次需求（新建系统或功能迭代）在 `spec/changes/{change-name}/` 建独立目录：

| 文件 | 内容 | 产品动作 |
|------|------|----------|
| `proposal.md` | 为什么做、做什么、不做什么 | 确认 Goal / Scope |
| `design.md` | 信息架构、数据模型、API、MVP、验收场景 | **重点评审，确认后再开发** |
| `tasks.md` | 可勾选实施清单 | 确认优先级与 MVP |
| `specs/{domain}/spec.md` | Delta 规格（ADDED / MODIFIED / REMOVED） | 核对需求变更完整性 |

### 四条硬性规则

1. **设计未确认，不写业务代码** — 未确认 `design.md` 前，禁止写 `src/app` 业务页和 `schema.ts` 业务表
2. **不直接改 specs/ 做新功能** — 必须走 `changes/` + Delta
3. **ME2AI 是输入，changes 是提案，specs 是结论**
4. **未 build 通过不得声称交付完成**

## Slide 8 — 二、专项成果 · @网站开发 Agent 与 Skill 编排

**布局**：成果展示（左说明 + 右 Agent / Skill 链路图）

### 标题

二、专项成果 - 成果展示 | @网站开发 Agent 与 Skill 编排

### Agent 定位

`@网站开发`（`.codebuddy/agents/engineering/网站开发.md`）是全栈 Web 应用交付的**主编排 Agent**：

- 按规范驱动流程，把需求变成**能本地运行、能持续迭代**的完整应用
- 适用：管理后台、业务平台、数据看板、协作门户等（新建或迭代均可）
- 内置原则：设计确认前不写代码；MVP 优先；交付前必须能演示、能构建

### Skill 编排链路

```
brainstorming（需求澄清）
    ↓ 产品确认
brainstorming（方案设计 → design.md）
    ↓ 产品确认
writing-plans（tasks.md）
    ↓
@网站开发 + executing-plans（全栈开发）
    ├── ui-ux-pro-max（UI/UX 设计 intelligence）
    ├── frontend-design（前端实现）
    ├── test-driven-development（测试驱动）
    └── verification-before-completion（交付验证）
    ↓
产品按验收场景逐条验收 → 归档
```

### 启动示例

```text
@网站开发

【项目名称】任务管理系统
【目标用户】团队负责人、成员
【核心功能】任务 CRUD、看板视图、首页统计
【MVP】第一版不做登录权限
【验收场景】
1. 首页能看到统计数据
2. 能新增、编辑、删除任务
3. 刷新后数据仍在

请先输出设计文档，我确认后再开发。
```

## 回写 PPT 指引
