## Slide 4 — 一、专项概述

**布局**：左文 + 右 OKR 表格

### 专项背景及意义

在处理一个总是出现信息丢失偶现的现场问题时，由于是偶现的，现场出现问题时再打开控制台，为时已晚，无法捕获问题发生时的控制台输出和 network 请求数据。这个问题对于前端开发，是一个普遍存在的问题。还有一些由于无法远程，只能让现场人员传递一些照片的场景，最让前端开发难受。

因此可以考虑设计一个浏览器的控制台日志和网络的抓取工具，可以监听到控制台和 network 请求信息。使用 AI 进行一些工具的开发创造，本来就是 AI 的强项，也借此功能验证 AI 工具开发能力。

### 整体目标及达成情况

开发出一个日志抓取工具：

- **现场人员**：安装后，抓取控制台和 network 网络请求信息，并导出抓取的日志到 json 文件
- **研发人员**：能够导入现场人员的日志文件，用来分析控制台和网络日志，定位原因

## Slide 8 — 二、专项成果 · 研发电脑

**布局**：成果展示（左说明 + 右多图：控制台/请求头/响应体）

### 标题

二、专项成果 - 成果展示 | 研发电脑

### 工具名称

Chrome Logger 工具（log-Viewer）：解决现场问题解读定位

### 应用示例

- **AI 应用使用角色**：前端研发
- **技术基础**：基于 JSON 解析完成问题分析功能，解决无法远程或无法抵达现场看到日志的痛点

### 技术分析定位

1. 日志查看器，支持现场录制 json 文件的导入和清空功能
2. 控制台日志，记录日志类型、时间、文本等信息
3. network 请求，支持查看完整的请求头，方便定位是否因为 token 或其他请求头导致的问题
4. network 请求，支持查看完整的请求体，方便定位因为请求数据异常导致的问题，抓取 json 格式的数据
5. network 请求支持响应体查看，记录 json 格式的响应数据，方便进行数据类型问题的定位
