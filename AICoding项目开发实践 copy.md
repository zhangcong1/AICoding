# AICoding 项目开发实践

> **面向对象**：产品经理  
> **分享人**：全栈开发  
> **背景**：公司已采购 AICoding（企业私有化 CodeBuddy），本文基于三个已落地系统的真实开发经验，说明产品经理如何参与、如何协作，以及如何用 AICoding 做出同类系统。

---

## 一、先建立共同认知：AICoding 是什么

**AICoding = 公司企业私有化部署的 CodeBuddy**，不是普通的「聊天式 AI」，而是一套可以**读代码、改代码、跑命令、调用工具、按方法论干活**的 AI 编程智能体。

| 对比维度 | 普通 AI 对话 | AICoding |
|----------|-------------|----------|
| 工作对象 | 文字回答 | 项目代码、文档、终端、数据库、API |
| 工作方式 | 一次性生成 | 多轮推理 + 自主执行 + 可验证 |
| 项目记忆 | 每次从零开始 | 通过 `CODEBUDDY.md`、Rules、Skills 记住项目约定 |
| 角色能力 | 通用助手 | 可加载产品经理、架构师、测试等专家 Agent |
| 工作方法 | 依赖个人 prompt 水平 | 可挂载 Superpowers 等工作流 Skill（先设计、再计划、再编码） |
| 企业特性 | 公网模型 | 私有化部署，代码不出域，符合公司安全合规要求 |

**一句话给产品经理**：AICoding 不是替开发写几段代码，而是把「需求澄清 → 方案设计 → 拆解计划 → 编码实现 → 测试验证 → 文档交付」整条链路加速。产品经理的价值，在于**把问题定义清楚、把边界划明白、把验收标准写具体**——这样 AICoding 才能稳定产出可用系统。

## 二、三个已落地系统：我们做了什么

以下三个系统均使用 AICoding 作为核心开发工具，由全栈开发主导，产品、业务方参与需求定义与验收。

### 2.1 大模型场景管理

**解决什么问题**

企业接入多个大模型后，Prompt、参数、版本、权限分散在各处，业务方无法自助配置，运维也难以统一治理。

**系统核心能力**

- 场景（Scenario）的创建、分类、版本管理
- Prompt 模板、变量、模型路由策略的可视化配置
- 场景发布 / 回滚、权限与租户隔离
- 调用日志、效果评估、A/B 对比

**AICoding 在其中的作用**

| 阶段 | AICoding 做了什么 | 产品经理参与点 |
|------|-------------------|----------------|
| 需求澄清 | 用 `brainstorming` 技能追问：谁用、怎么用、成功标准是什么 | 提供用户角色、业务场景、合规约束 |
| 方案设计 | 输出数据模型（场景/版本/模板/权限）、API 清单、页面结构 | 评审信息架构，确认「场景」与「模板」的业务语义 |
| 实施计划 | 拆成：后端 CRUD → 版本引擎 → 前端配置台 → 权限 → 日志 | 确认优先级：先做「能配能用」，再做「能评能管」 |
| 编码实现 | 生成 Spring Boot / Vue 模块、数据库迁移、接口文档 | 验收原型页面，反馈交互是否符合业务语言 |
| 测试验证 | 自动生成 API 测试、边界用例、权限用例 | 提供 3～5 个真实业务场景做验收脚本 |

**关键经验**

- 「场景」是产品语言，「Prompt 模板 + 模型参数 + 路由规则」是技术实现——**先在 AICoding 里对齐词汇表**，再写代码。
- 版本管理是高频变更点，产品必须定义：**什么算新版本、谁可以发布、回滚影响范围**。

---

### 2.2 代码智能解析

**解决什么问题**

研发资产（代码库、依赖、接口、调用链）难以被非开发人员理解；新人上手慢，架构评审缺乏结构化输入。

**系统核心能力**

- 代码仓库接入与索引（语言识别、目录结构、模块划分）
- AST / 依赖 / 调用链解析
- 自动生成模块说明、接口文档、变更影响分析
- 自然语言问答：「这个接口被谁调用」「改这里会影响哪些模块」

**AICoding 在其中的作用**

| 阶段 | AICoding 做了什么 | 产品经理参与点 |
|------|-------------------|----------------|
| 需求澄清 | 区分「给开发用」还是「给架构/管理用」，明确输出形态 | 定义目标用户、使用频率、可接受的解析延迟 |
| 方案设计 | 设计解析流水线：Clone → Parse → Index → Query；选型 Tree-sitter / LSP | 确认 MVP：先支持 Java + Vue，还是先单语言 |
| 实施计划 | 分模块：解析引擎 → 索引存储 → 查询 API → 前端展示 → 问答层 | 排期：先「看得懂结构」，再「问得出来」 |
| 编码实现 | 编写解析器、Graph 存储、REST API、可视化页面 | 验收：拿真实项目试跑，看输出是否「人话化」 |
| 测试验证 | 用真实仓库做回归，对比解析准确率 | 提供 2～3 个典型仓库作为黄金样本 |

**关键经验**

- 这类系统的「产品体验」= **解析结果是否可信、是否可解释**。PM 要验收的不是「有没有结果」，而是「结果能不能支撑决策」。
- AICoding 本身就在写「让 AI 理解代码」的系统——开发过程中会大量用到 `@engineering-software-architect`、`@engineering-code-reviewer` 等 Agent 做自检。

---

### 2.3 智能体人际协作平台

**解决什么问题**

单一 AI 对话无法支撑复杂任务；多人、多角色、多步骤协作缺乏统一工作台，人工审批与 AI 执行之间缺少衔接。

**系统核心能力**

- 多 Agent 角色定义（产品、开发、测试、运维等）
- 任务编排：串行 / 并行 / 人工卡点
- 人机协作：AI 提案 → 人工确认 → AI 继续执行
- 会话历史、产物归档、权限与审计

**AICoding 在其中的作用**

| 阶段 | AICoding 做了什么 | 产品经理参与点 |
|------|-------------------|----------------|
| 需求澄清 | 梳理「谁触发任务、谁审批、谁只看结果」 | 定义协作流程：例如 PRD 评审、代码审查、发布审批 |
| 方案设计 | 设计 Agent 注册表、Task DAG、Human-in-the-loop 节点 | 画业务流程图，标注哪些步骤必须人工介入 |
| 实施计划 | 先后端编排引擎 → Agent 调度 → 前端工作台 → 审计日志 | 确认 MVP：先「单任务多 Agent」，再「多任务编排」 |
| 编码实现 | 集成 MCP、Sub-Agent、工作流 YAML（`workflow-runner`） | 配置首批 Agent 角色与 Prompt，参与联调 |
| 测试验证 | 模拟完整协作链路，验证状态机与权限 | 走查 1～2 条端到端业务流 |

**关键经验**

- 这个平台本质上是在**复刻 AICoding 自身的工作方式**——产品要先理解：Agent、Skill、Task、审批，分别对应什么用户动作。
- 产品经理可以直接参与 Agent 角色定义（`.codebuddy/agents/` 目录下的专家配置），**把产品方法论写进 Agent，而不是只写 PRD 扔给开发**。

## 三、三个系统的共性模式

尽管业务不同，三个系统都遵循同一套 **AICoding 项目开发模式**：

```
┌─────────────────────────────────────────────────────────────┐
│  产品经理：问题定义 + 边界 + 验收标准 + 业务语义对齐          │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  AICoding + Superpowers：brainstorming → writing-plans     │
│  → executing-plans → TDD → review → verification           │
└───────────────────────────┬─────────────────────────────────┘
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  交付物：可运行系统 + API 文档 + 设计文档 + 测试证据          │
└─────────────────────────────────────────────────────────────┘
```

| 共性 | 说明 |
|------|------|
| 先设计后编码 | 再「简单」的需求也走 brainstorming，避免 AI 直接开写返工 |
| 项目级记忆 | `CODEBUDDY.md` 记录技术栈、命名规范、架构约定，全项目共享 |
| 专家 Agent 分工 | 产品、架构、前端、后端、测试等 Agent 按场景调用 |
| 工作流 Skill | Superpowers 保证 TDD、调试、审查、验证有章可循 |
| 人机协作节点 | 方案确认、发布审批、验收测试——产品必须在环 |

## 四、产品经理在 AICoding 项目中的四个角色

### 4.1 问题定义者（最重要）

不要上来就说「做一个 XXX 功能」，而要说：

- **谁**在用？什么场景？
- **现在**怎么做的？痛点是什么？
- **成功**长什么样？怎么度量？
- **不能**做什么？合规 / 性能 / 安全边界？

**推荐话术（直接对 AICoding 说）**

```text
我要做一个 [系统名称]。
目标用户是 [角色]，他们在 [场景] 下需要 [能力]。
当前痛点是 [具体问题]。
成功标准是 [可量化指标]。
约束条件：[技术栈 / 合规 / 时间 / 不能做的范围]。
请先不要写代码，帮我做需求澄清和方案对比。
```

这会触发 `brainstorming` 技能，AI 会逐一追问，正好补齐 PRD 所需信息。

---

### 4.2 方案评审者

AICoding 产出设计文档后，产品经理重点评审：

| 评审项 | 关注点 |
|--------|--------|
| 信息架构 | 菜单、实体、关系是否符合业务语言 |
| 用户路径 | 主流程是否 3～5 步内完成 |
| 权限模型 | 谁能看、谁能改、谁能发布 |
| 边界情况 | 空数据、失败、并发、回滚 |
| MVP 范围 | 第一版做什么、不做什么 |

**你不需要懂代码**，但需要问：「业务方能不能看懂、用起来顺不顺、出了错怎么办」。

---

### 4.3 验收负责人

AICoding 完成开发后会走 `verification-before-completion`——没有跑过测试证据，不能声称「做完了」。

产品经理的验收清单：

- [ ] 主流程走通（附操作步骤）
- [ ] 3～5 个真实业务场景可复现
- [ ] 权限与角色行为符合预期
- [ ] 异常提示是「人话」，不是堆栈信息
- [ ] 核心 API / 页面有文档或说明

---

### 4.4 Agent 配置参与者（进阶）

在智能体协作类项目中，产品经理可以直接参与 `.codebuddy/agents/product/` 下的角色定义，例如：

- `product-manager`：PRD、路线图、优先级
- `product-trend-researcher`：竞品与市场调研
- `product-feedback-synthesizer`：用户反馈分析

**含义**：把「产品经理怎么思考」写成 Agent 配置，让平台里的 AI 角色与真实产品流程一致。

## 五、AICoding 项目标准配置（建议）

每个新项目建议在根目录建立 `.codebuddy/` 结构：

```
项目根目录/
├── CODEBUDDY.md              # 项目记忆：技术栈、规范、业务背景
├── AICoding项目开发实践.md    # 本文档（团队共识）
└── .codebuddy/
    ├── skills/               # 工作方法论（如 Superpowers）
    ├── agents/               # 专家角色（产品/架构/开发/测试…）
    │   ├── product/
    │   ├── design/
    │   ├── engineering/
    │   └── testing/
    └── rules/                # 团队编码与文档规范（可选）
```

| 目录 | 产品经理需要知道什么 |
|------|----------------------|
| `CODEBUDDY.md` | 项目的「长期记忆」，应包含业务背景、用户角色、术语表 |
| `.codebuddy/skills/` | AI 的工作方法（先设计、TDD、审查…），一般研发维护 |
| `.codebuddy/agents/` | 可调用的专家角色，产品可参与 `product/` 分类 |
| `.codebuddy/rules/` | 强制规范（命名、安全、文档格式），产品可提需求 |

## 六、用 AICoding 做新系统的标准流程

### Step 1：启动会话，对齐上下文

```text
请先阅读 CODEBUDDY.md 和项目现有结构。
我要启动一个新系统：[系统名称]。
请先理解项目背景，再开始需求澄清。
```

### Step 2：需求澄清（brainstorming）

- AI 会逐条提问，产品经理逐条回答
- 产出：设计规格文档（通常保存在 `docs/superpowers/specs/`）
- **产品动作**：确认方案，明确「做 / 不做」

### Step 3：实施计划（writing-plans）

- AI 把设计拆成可执行任务：文件、接口、页面、测试
- **产品动作**：确认优先级，砍 MVP 范围

### Step 4：开发执行（executing-plans + TDD）

- 开发（或 AICoding 自主）按计划实现
- **产品动作**：中期看原型，及时纠偏，避免做完才发现理解错了

### Step 5：审查与验证

- 代码审查 + 测试证据 + 产品验收
- **产品动作**：按验收清单签字，或列出修改项

### Step 6：收尾

- 合并、部署、文档归档
- **产品动作**：更新用户说明、培训材料、上线公告

## 七、三类系统的启动模板（可直接复用）

### 模板 A：配置管理平台（参考：大模型场景管理）

```text
我要做一个 [XX] 配置管理平台。
核心实体是 [实体名]，需要支持 CRUD、版本管理、发布/回滚、权限隔离。
用户角色：[管理员 / 业务配置员 / 只读查看者]。
第一版只做 [MVP 范围]，不做 [明确排除项]。
请先输出：数据模型、API 列表、页面结构、权限矩阵。
```

### 模板 B：分析洞察平台（参考：代码智能解析）

```text
我要做一个 [XX] 智能分析平台。
输入是 [数据源]，输出是 [结构化结果 + 自然语言问答]。
目标用户是 [角色]，他们需要通过它 [完成什么决策]。
请先定义：解析流水线、存储方案、查询接口、前端展示形态。
MVP 先支持 [一种数据源 / 一种语言 / 一种输出格式]。
```

### 模板 C：协作编排平台（参考：智能体人际协作平台）

```text
我要做一个 [XX] 人机协作平台。
支持 [N] 种 Agent 角色，任务可 [串行/并行] 执行。
必须在 [节点] 插入人工审批。
请先输出：Agent 清单、任务状态机、审批流程、审计日志设计。
第一版先跑通一条完整业务流：[具体流程描述]。
```

## 八、产品经理 vs 开发：分工边界

| 事项 | 产品经理 | 全栈开发 + AICoding |
|------|----------|---------------------|
| 为什么要做 | ✅ 主导 | 参与澄清 |
| 做什么 / 不做什么 | ✅ 主导 | 评估可行性 |
| 用户流程与信息架构 | ✅ 主导 | 实现与优化 |
| 技术选型 | 参与决策 | ✅ 主导 |
| 数据模型与 API | 评审业务语义 | ✅ 主导 |
| 编码实现 | — | ✅ 主导 |
| Agent / Skill 配置 | 参与产品类 Agent | ✅ 主导 |
| 测试与上线 | ✅ 验收 | ✅ 执行 |
| 文档与培训 | ✅ 用户向文档 | ✅ 技术文档 |

**避免的两个极端**

1. **产品不管，只说「帮我做一个系统」** → AI 会猜需求，返工成本高  
2. **产品过度介入实现细节** → 限制 AI 和开发发挥，效率反而下降

## 九、常见问题（产品经理视角）

**Q1：AICoding 会不会取代开发？**  
不会。AICoding 放大的是开发效率；**问题定义、取舍、验收、跨部门对齐**仍然需要人和产品经理。

**Q2：产品需要会写 prompt 吗？**  
需要会「说清楚问题」，不需要会调模型参数。按本文模板表达即可；复杂 Prompt 由开发和 Agent 配置承担。

**Q3：需求变更怎么办？**  
在 AICoding 里明确说「范围变更：原 X 不做，改为 Y」，让它更新设计文档和计划，不要口头改一半、代码改一半。

**Q4：怎么判断 AICoding 产出是否可信？**  
看证据：测试是否跑过、演示是否可复现、文档是否与代码一致。没有证据的「已完成」不可验收。

**Q5：三个系统之间有什么可复用能力？**  
| 复用层 | 内容 |
|--------|------|
| 基础平台 | 用户 / 角色 / 权限 / 审计日志 |
| 配置引擎 | 版本管理、发布、回滚（场景管理 → 其他配置类系统） |
| 解析引擎 | 索引、检索、问答（代码解析 → 文档/知识库类系统） |
| 编排引擎 | Agent 调度、人工卡点（协作平台 → 其他流程类系统） |

## 十、给产品经理的行动建议

### 本周就能做的

1. 读一遍 `CODEBUDDY.md`，了解当前项目约定  
2. 用 **模板 A/B/C** 之一，在 AICoding 里试一次「只澄清需求、不写代码」  
3. 参加一次开发会话的中期原型评审，而不是等全部做完  

### 下一个项目应前置的

1. 与开发共建 `CODEBUDDY.md` 中的**业务术语表**和**用户角色说明**  
2. 在 PRD 中增加「验收场景」章节（3～5 条，可执行）  
3. 明确 MVP 边界，写清「第一版不做什么」  

### 长期可参与的

1. 共同维护 `.codebuddy/agents/product/` 下的产品 Agent  
2. 把反复出现的评审标准沉淀为 `.codebuddy/rules/`  
3. 推动团队统一 Superpowers 工作流，减少「AI 想到哪写到哪」

## 十一、总结

| 结论 | 说明 |
|------|------|
| AICoding 是工具，不是魔法 | 输入质量决定输出质量 |
| 产品经理的核心价值不变 | 定义问题、对齐预期、验收结果 |
| 三个已落地系统证明路径可行 | 配置管理、智能解析、协作编排均可复制 |
| 复制路径 = 标准流程 + 项目配置 + 明确分工 | 本文即团队可复用的起点 |

**最后一句话**：  
用 AICoding 做系统，不是「把需求扔给 AI 等结果」，而是**产品定义方向、AI 加速实现、人做关键判断**——三个已落地系统，都是这样做出来的。
